escape 、encodeURI 、encodeURIComponent的区别与应用场景

首页 / 新闻资讯 / 正文

背景
最近，在公司封装短信SDK时，遇到对请求参数进行 encodeURIComponent的场景。结合在做微信分享时，微信分享接口也要求传到服务器端的请求参数，必须进行 encodeURIComponent编码。于是，十分困惑，为什么需要对链接或请求参数进行 encodeURIComponent编码？何时需要进行编码？encodeURIComponent与encodeURI、escape又有什么区别呢？

三种编码方式与应用场景
1）escape()
已废弃
该特性已经从 Web 标准中删除，虽然一些浏览器目前仍然支持它，但也许会在未来的某个时间停止支持，请尽量不要使用该特性。

作用：escape() 函数可对字符串进行编码，这样就可以在所有的计算机上读取该字符串。

说明：该方法不会对 ASCII 字母和数字进行编码，也不会对下面这些 ASCII 标点符号进行编码： * @ - _ + . / 。其他所有的字符都会被转义序列替换。

与此方法相对的方法 —— unescape()，作用是对escape()编码的字符串进行解码。

由于，此方法已被废弃。请勿在生产环境使用此方法。

2）encodeURI()
作用：对字符串作为URI进行编码。（对整个URL进行编码，因此，不会对网址中特殊含义符号"; / ? : @ & = + $ , #"，不会进行编码）

说明：该方法不会对 ASCII 字母和数字进行编码，也不会对这些 ASCII 标点符号进行编码： - _ . ! ~ * ' ( ) 。
该方法的目的是对 URI 进行完整的编码，因此对以下在 URI 中具有特殊含义的 ASCII 标点符号，encodeURI() 函数是不会进行转义的：;/?:@&=+$,#

应用场景：当期望获取一个可用的URL地址时，使用此方法进行编码。

例如，

encodeURI("http://www.example.org/a file with spaces.html")

# 编码结果
http://www.example.org/a%20file%20with%20spaces.html

# 若使用 encodeURIComponent进行编码，则无法得到一个可正常工作的链接。
# 可用将编码后的地址，粘贴到浏览器进行验证。
encodeURIComponent("http://www.examp。e.org/a file with spaces.html")

# 编码结果
http%3A%2F%2Fwww.example.org%2Fa%20file%20with%20spaces.html
3）encodeURIComponent()
作用：encodeURIComponent()是对统一资源标识符（URI）的组成部分进行编码的方法，而不用对整个URL进行编码。
在 encodeURI() 中不被编码的符号"; / ? : @ & = + $ , #"，encodeURIComponent()都会进行编码。（可以观察到，都是一些常见的网络协议标志符——例如，“/@?&#'”）

应用场景：当需要对URL的参数进行编码时，使用encodeURIComponent()。

例如，

# 对部分参数进行编码
var p1 = encodeURIComponent("http://example.org/?a=12&b=55")
# 拼接参数
var url = "http://example.net/?param1=" + p1 + "&param2=99";

# 最终url的地址
http://example.net/?param1=http%3A%2F%2Fexample.org%2F%Ffa%3D12%26b%3D55&param2=99

总结
对于开头提到的问题进行解答：

为什么需要对URL进行编码？
统一各端（操作系统、浏览器等）对URL编码方式。由于RFC 1738，没有对「URL编码」制定具体的编码规则。不同的操作系统、不同的浏览器、不同的网页字符集，可能对同一地址产生不同的编码。为了保证客户端发送给服务端的请求，编码格式相同。因此，在客户端使用JavaScript的编码方式统一对URL进行编码。这样，服务端接受的请求的编码方式就是相同的。想要了解更多的同学，可以阅读阮一峰老师的《关于URL编码》这篇文章。

何时使用encodeURIComponent进行编码
当需要对URL组成部分进行编码时，使用 encodeURIComponent。例如，微信JS-SDK分享朋友圈接口，需要将分享的链接作为参数回传给微信的服务器。此时，分享链接作为参数，因此需要进行编码处理。
